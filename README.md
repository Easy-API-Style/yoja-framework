Yoja is a **full-stack framework** — backend in **pure Java**, frontend in **JavaScript / CSS / HTML**.

The backend runs on [Vert.x](https://vertx.io/): non-blocking HTTP server, WebSocket, sessions, reverse proxy. The frontend runs on **native browser APIs** with no build toolchain, no `node_modules`, and no virtual DOM. It follows the **React component pattern** — each section of the page is driven by an ES6 class controller that receives a handle to its DOM subtree — but without the VDOM, the build step, or the transpiler.

## Why Yoja?

### Frontend

| | React / Vue / Angular | yoja-web |
|---|---|---|
| Build step required | Yes | No |
| `node_modules` | Yes (hundreds of MB) | None |
| Virtual DOM | Yes | No (direct DOM) |
| CSS scoping | Build-time (class hashing) | Runtime (native Shadow DOM) |
| HTML / CSS / JS separation | No (JSX mixes all three) | Yes (one file per concern) |
| CSS Flexbox support | Yes | Yes (native, no interference) |
| i18n built-in | No | Yes (XML per section) |
| HTTP client built-in | No | Yes |
| WebSocket client built-in | No | Yes |
| Responsive service built-in | No | Yes |
| Gradual adoption | Hard (full rewrite) | Easy (section by section) |

**No build toolchain.** Serve your `.html`, `.css`, and `.js` files directly — no Webpack, no Vite, no Babel, no TypeScript compiler. Any static HTTP server works.

**Strict separation of concerns.** HTML defines structure, CSS defines appearance, JavaScript defines behaviour — each in its own file. No JSX mixing markup with logic, no `<style>` blocks inside components, no CSS-in-JS.

**CSS Flexbox works natively.** Each section is a real `<div>` in the DOM, so `display: flex`, `flex-direction`, `gap`, and all other layout properties apply exactly as in plain HTML — no framework interference.

**Native Shadow DOM isolation.** Each section is a shadow root: its CSS is physically scoped by the browser via `adoptedStyleSheets`. No generated class-name hashes, no post-processing — styles from one section never leak into another.

**Gradual adoption.** Drop yoja-web into an existing HTML page one section at a time. No need to rewrite an entire application before you can start using it.

### REST services

| | Spring Boot | Vert.x (raw) | yoja-framework |
|---|---|---|---|
| Event loop model | No (thread-per-request) | Yes | Yes |
| Non-blocking I/O | No (Servlet by default) | Yes | Yes |
| Declarative routing | Yes (annotations) | Manual | Yes (fluent builder) |
| Built-in WebSocket | No (plugin) | Yes | Yes |
| Session management | Yes (Spring Session) | Manual | Yes (built-in) |
| Request/response interceptors | Yes (filters) | Manual | Yes (built-in) |
| SSL with hot-reload | No | Manual | Yes (zero downtime) |
| Reverse proxy built-in | No | No | Yes |
| No reflection / annotation scanning | No | Yes | Yes |

**Non-blocking by default — Multi-Reactor pattern.** The server is built on the Vert.x event loop, which implements the [Reactor pattern](https://en.wikipedia.org/wiki/Reactor_pattern): a small number of threads handles thousands of concurrent connections by dispatching events rather than blocking. No thread-per-request overhead, no blocking I/O. Worker threads are available for CPU-intensive or blocking tasks without stalling the event loop.

**Fluent, code-only configuration.** Routes, middleware, sessions, WebSocket endpoints, and SSL are all wired in plain Java with a builder API — no XML, no annotations, no classpath scanning at startup.

**Global request and response interceptors** let you enforce authentication, add response headers, or transform payloads in one place, without duplicating logic across every route.

**SSL hot-reload** allows updating a certificate at runtime without restarting the server — zero downtime in production.

**Reverse proxy included.** A single `ReverseProxyServer` can terminate SSL and route traffic to multiple backend instances with host/path rules, path rewriting, and load balancing — no external Nginx or HAProxy required for simple deployments.

### Testing

| | JUnit + Mockito | Selenium (standalone) | yoja-selenium |
|---|---|---|---|
| Real browser tests | No | Yes (manual setup) | Yes (automatic setup) |
| HTTP server started automatically | No | No | Yes |
| Browser started automatically | No | No | Yes |
| Test URL resolution | Manual | Manual | Yes (`HttpServerContext`) |
| JUnit 6 integration | Yes | No | Yes |
| Multi-browser support | — | Yes | Yes |

**Real browser, real server, zero setup.** `YojaSeleniumExtension` starts a `yoja-http-server` instance and a Chrome/firefox/edge browser before each test, injects them as parameters, and tears them down after — no `@BeforeEach` boilerplate, no port management, no `WebDriver` lifecycle code.

**Tests run against the real stack.** There are no mocks, no in-memory stubs, no simulated HTTP. The browser navigates to the actual server serving the actual frontend — the same code path that runs in production.

**URL helpers included.** `HttpServerContext` gives you `ctx.url("/api/users")` with the correct host, port, and context path already filled in — no hardcoded localhost strings.

### AI-assisted development

Yoja's design choices happen to align closely with what makes an AI coding assistant most effective.

**Code-only configuration — nothing hidden from the model.** Routes, middleware, sessions, SSL, and WebSocket endpoints are all wired in plain Java with a builder API. There is no XML to cross-reference, no annotation processor to understand, and no classpath-scanning magic. An AI can read the source and reason about the full application state from a single pass over the code.

**Explicit over implicit — generated code is predictable.** Because Yoja has no convention-over-configuration layer and no framework annotations, there is no hidden rule that silently changes behaviour. What an AI writes is exactly what executes, which makes it straightforward to verify the output and trace bugs.

**Strict file separation — targeted edits with no side effects.** Each section has one `.js` file for behaviour, one `.css` file for appearance, and one `.xml` file for translations. An AI can rewrite the CSS without touching the logic, or add a translation key without opening the controller. File boundaries match concern boundaries.

**No build toolchain for the frontend — instant feedback.** AI-generated JavaScript runs directly in the browser with no compilation, no transpilation, and no bundler configuration. There is no pipeline step that could silently transform or reject the generated output.

**Small, focused API surfaces — lower hallucination risk.** Each module exposes a narrow, well-documented API. An AI has fewer patterns to choose from and fewer opportunities to invent plausible-but-wrong method names or configuration keys.

**Real-stack tests — no mock contracts to maintain.** `yoja-selenium` runs the browser against the real server. An AI writing a test does not need to maintain fake HTTP responses, synchronise stub state with production code, or understand a mocking framework. The assertion targets real behaviour, and a passing test means the full stack works.

## Architecture

A typical Yoja application is structured as follows:

```
Browser
  └── yoja-web              (HTML + CSS + JS, served as static files)
        └── HTTP / WebSocket
              └── yoja-http-server   (REST API + static file server)
                    ├── yoja-core    (lifecycle, workers, async utilities)
                    └── yoja-http-client  (calls to third-party APIs)

Production
  └── yoja-reverse-proxy    (routes traffic to one or more yoja-http-server instances)

Testing
  └── yoja-selenium         (end-to-end browser tests with JUnit 6)
```

All server-side modules depend on `yoja-core`. The browser-side module (`yoja-web`) is independent — it is pure JavaScript loaded directly in the browser.

## Example Application

The [`yoja-blueprint-kanban`](https://github.com/Easy-API-Style/yoja-blueprint-kanban) module is a full-stack task management application that demonstrates the main features of the framework end to end: HTTP server, REST API, WebSocket for real-time updates, session management, outbound HTTP client, background workers, and the `yoja-web` frontend (Kanban board, login, i18n, URL-parameter navigation).

It is the fastest way to see how all the modules fit together in a real project, and it comes with a complete end-to-end test suite driven by a headless Chrome browser via `yoja-selenium`.

## Building a Website: Step by Step

### 1. Serve the frontend

`yoja-http-server` can serve static files (HTML, CSS, JS) from the classpath. This is how the browser downloads the `yoja-web` application.

```java
HttpRouter router = 
    HttpRouter.builder()
              // serves classpath resources as static files
              .webResource(WebApp.jar("com.example.webapp"), "/*") 
              .build();

HttpServer.builder(router, 443)
          .ssl(Path.of("/etc/letsencrypt/live/example.com/privkey.pem"),
               Path.of("/etc/letsencrypt/live/example.com/fullchain.pem"))
          .start()
          .onSuccess(server -> System.out.println("Server started on port " + server.port()));
```

The classpath package `com.example.webapp` contains your `index.html` and all `yoja-web` assets.

### 2. Add REST API routes

Define your API endpoints using `HttpRouter`. Handlers run on the Vert.x event loop — use `Worker` for any blocking logic.

```java
HttpRouter router = 
    HttpRouter.builder()
              // serves classpath resources as static files
              .webResource(WebApp.jar("com.example.webapp"), "/*")  
              // serves classpath webservices as REST services
              .webService(HttpMethod.GET, "/api/users", routing -> {
                  routing.response().send(new JsonArray().add("Alice").add("Bob"));
              })
              .webService(HttpMethod.POST, "/api/users", routing -> {
                  JsonObject body = routing.request().bodyAsJsonObject();
                  // process body...
                  routing.response().statusCode(201).send();
              })
             .build();

HttpServer.builder(router, 443)
          .ssl(Path.of("/etc/letsencrypt/live/example.com/privkey.pem"),
               Path.of("/etc/letsencrypt/live/example.com/fullchain.pem"))
          .start();
```

### 3. Write the frontend

In your `webapp/index.html`, include the `yoja-web` script and declare your first section:

```html
<!DOCTYPE html>
<html>
<head>
    <meta http-equiv="Content-Type" content="text/html;charset=utf-8"/>
    <meta http-equiv="encoding" content="utf-8"/>
    <meta name="viewport" content="width=device-width, initial-scale=1"/>

    <script type="module" src="/com/easygoingapi/yoja/web/YojaWeb-1.0.0.js"></script>
</head>
<body>
    <div yw-controler="./sections/UserList/UserListControler.js"
         yw-css="./sections/UserList/userList.css"
         yw-language="./sections/UserList/userList.lang.xml">

         ...
    </div>
</body>
</html>
```

Each section is an isolated component with its own HTML, CSS, and JavaScript file. CSS Flexbox works seamlessly — apply `display: flex` directly on the section container or any element inside it.

```
webapp/
  index.html
  YojaWeb.conf.js            ← global config (declares the app version — see below)
  sections/
    UserList/
      UserListControler.js   ← behaviour
      userList.css           ← appearance (scoped via Shadow DOM)
      userList.lang.xml      ← i18n
```

`YojaWeb.conf.js` exposes a `version` property that is **the cache-busting mechanism** of the framework: it is appended as `?version=<value>` to every HTML, CSS and JS resource loaded by `yoja-web` (controllers, stylesheets, included HTML, language XML, slot templates). Bump this string on each new release of your application and every browser is forced to re-download the assets instead of serving stale copies from its HTTP cache — without it, users would keep running old code after a deploy until their cache expires naturally. See [yoja-web/README.md](yoja-web/README.md#yojaweb-configuration) for the full configuration reference.

HTML, CSS, and JavaScript are strictly separated: each file has a single responsibility, with no JSX, no `<style>` blocks inside components, and no CSS-in-JS.

```js
// UserListControler.js
export default class UserListControler {
    constructor(section) {
        section.pageReady(() => {
            section.firstTag('ul').textContent = 'Loading...';
        });
    }
}
```

### 4. Call the API from the browser

Use the built-in `yoja-web` HTTP client to call your backend API:

```js
import { userListService } from '/UserListService.js';
const http = yojaWebApi.httpClient;

export default class UserListControler {
    constructor(section) {
        section.pageReady(() => {
            http.get('/userRoles').then(v => {
                const users = userListService.getUsers();
                // populate the DOM...
            });
        });
    }
}
```

### 5. Add a reverse proxy for production

In production, `yoja-reverse-proxy` sits in front of one or more `yoja-http-server` instances. It handles SSL termination, load balancing, and routing by host or path.

```java
ReverseProxyServer.builder()
                  .port(443)
                  .ssl(Path.of("/etc/letsencrypt/live/example.com/privkey.pem"),
                       Path.of("/etc/letsencrypt/live/example.com/fullchain.pem"))
                  .rule(ReverseProxyRule.builder()
                                        .host("www.example.com")
                                        .targetHost("localhost")
                                        .targetPort(8080)
                                        .build())
                  .build()
                  .start();
```

### 6. Write end-to-end tests

`yoja-selenium` integrates Selenium WebDriver with JUnit 6 to run browser tests against a real HTTP server.

```java
    @BeforeEach
    void setUp() {
        selenium = SeleniumService.newInstance(Browser.builderBrowser.CHROME)
                                                      .mode(Browser.Mode.HEADLESS)
                                                      .build());
    }

    @Test
    void loginPageIsDisplayed() {
        selenium.getHttpPage(appUrl("/index.html"));
        WebElement list = selenium.firstTag("ul");
        assertNotNull(list);
    }
```

## Gradle Setup

Add only the modules you need. All modules require `yoja-core`.

```groovy
dependencies {
    // Required for all server-side modules
    implementation 'com.easygoingapi:yoja-core:VERSION'
    // HTTP server (REST API + static files + WebSocket)
    implementation 'com.easygoingapi:yoja-http-server:VERSION'
    // HTTP client (call external APIs or other services)
    implementation 'com.easygoingapi:yoja-http-client:VERSION'
    // Reverse proxy (production routing)
    implementation 'com.easygoingapi:yoja-reverse-proxy:VERSION'
    // End-to-end testing
    testImplementation 'com.easygoingapi:yoja-selenium:VERSION'
}
```

`yoja-web` is a JavaScript library — include it as a static file in your `webapp/`, no Gradle dependency needed.

## Module Details

### yoja-core

The foundation of the framework. Every server-side module depends on it.

- `YojaApp` — starts and stops the Vert.x instance
- `Worker` — offloads blocking work to background threads (single-thread or thread pool)
- `Timer` — schedules recurring or one-shot tasks
- `FutureUtil` — bridges async `Future`s with blocking worker code (`await`, `awaitValue`)
- HTTP utilities — `HttpUrl`, `HttpHeader`, `HttpCookie`, `HttpParameter`, `HttpEncoding`
- General utilities — `StringUtil`, `TimeUtil`, `ResourceUtil`, `PathUtil`, `ProcessUtil`

See [yoja-core/README.md](yoja-core/README.md) for full documentation.

---

### yoja-http-server

The HTTP and WebSocket server. Built on Vert.x, it handles routing, middleware, static resources, cookies, and SSL.

- `HttpServerApp` — declare routes (`GET`, `POST`, `PUT`, `DELETE`, `WebSocket`), middleware, and static resources
- `HttpServer` — start and stop the server
- `HttpCertificate` — `NONE`, `SSL`, `SELF_SIGNED`
- Request/response interceptors — run logic before or after every request
- Cookie management — read and write cookies with full `SameSite`, `HttpOnly`, `Secure` support

See [yoja-http-server/README.md](yoja-http-server/README.md) for full documentation.

---

### yoja-http-client

The HTTP and WebSocket client. Used by the backend to call external APIs or communicate between services.

- `HttpEngine` — manages the underlying Vert.x `WebClient`
- `HttpClient` — targets a host/port and sends `HttpGet` / `HttpPost` requests; returns `Future<HttpResponse>`
- `HttpGet` / `HttpPost` — request builders with query parameters, headers, cookies, and body
- `HttpResponse` — read status, headers, cookies, and body (JSON, text, binary)
- `WebSocketEngine` / `WebSocketClient` — open WebSocket connections and exchange messages

See [yoja-http-client/README.md](yoja-http-client/README.md) for full documentation.

---

### yoja-web

The browser-side framework. Served as a single JavaScript file — no `npm install`, no build step, no bundler.

- Section-based component architecture — each section is a Shadow DOM root with scoped CSS
- `yw-controler` — attach a JavaScript controller to a section
- `yw-css` — load scoped CSS into the section's shadow root via `adoptedStyleSheets`
- `yw-language` — i18n via XML files
- Built-in HTTP and WebSocket client
- Client-side routing
- Local and session storage utilities
- Responsive breakpoint service

CSS Flexbox works seamlessly: each section is a real `<div>` element, so `display: flex` and all flex properties apply without any interference from the framework. HTML, CSS, and JavaScript stay in separate files — no JSX, no template syntax, no CSS-in-JS.

See [yoja-web/README.md](yoja-web/README.md) for full documentation.

---

### yoja-reverse-proxy

The production reverse proxy. Sits in front of one or more `yoja-http-server` instances and handles traffic routing.

- `ReverseProxyServer` — starts the proxy on a configured port
- `ReverseProxyRule` — routes requests by host or path to a target host/port
- Path rewriting — strip or rewrite the URL prefix before forwarding
- Load balancing — distribute traffic across multiple backend instances
- SSL termination — accept HTTPS from clients, forward plain HTTP to backends
- Admin API — add, update, or remove routing rules at runtime without restarting

See [yoja-reverse-proxy/README.md](yoja-reverse-proxy/README.md) for full documentation.

---

### yoja-selenium

End-to-end testing module. Integrates Selenium WebDriver with JUnit 6 to run browser tests against a real `yoja-http-server` instance.

- `YojaSeleniumExtension` — JUnit 6 extension that starts a browser and an HTTP server for each test
- `Browser` — wraps `WebDriver` with convenience methods
- `BrowserConfig` — configure Chrome options, headless mode
- `HttpServerContext` — provides the base URL of the test server
- `HttpUrlBuilder` — constructs test URLs with paths and query parameters

See [yoja-selenium/README.md](yoja-selenium/README.md) for full documentation.
